# ADR-0008 分离 Liveness 与 Readiness

## 状态

已接受（2026-07-16）

## 背景

原有 `/health` 接口固定返回 `ok`，无法反映数据库是否可用。同时，应用在导入 `app.main` 时主动连接数据库；数据库不可用会导致 FastAPI 无法启动，健康检查本身也无法访问。

一个运行中的应用进程和一个可以接收业务流量的应用不是同一个状态，需要分别判断。

## 决策

- 提供 `GET /health/live`，只检查 FastAPI 进程能否响应，不访问外部依赖。
- 提供 `GET /health/ready`，检查 MySQL 等必要依赖是否可用。
- Readiness 正常时返回 HTTP 200，依赖不可用时返回 HTTP 503。
- 禁止在 Python 模块导入阶段连接数据库。
- 数据库探测由数据库层执行，依赖状态由 Health Service 组合，Router 只负责 HTTP 协议。
- Jenkins 蓝绿发布使用 `/health/ready` 判断新版本是否可以接收流量。
- 未来 Kubernetes 分别使用 liveness 和 readiness 探针。

## 原因

- 数据库短暂故障时，应用仍可提供 liveness 并暴露真实运行状态。
- Readiness 可以阻止流量进入依赖尚未就绪的实例。
- 模块导入不产生数据库连接副作用，更容易启动、测试和排查问题。
- 分层后可以在未来加入 Redis、对象存储等依赖检查，而不把基础设施逻辑写进 Router。

## 影响

- 应用启动成功不再代表数据库一定可用。
- 运维和发布系统必须根据目的选择正确端点。
- 健康检查会被频繁调用，成功探测不记录普通日志，避免日志噪声。

## 未采用方案

- 保留单一 `/health`：无法区分进程存活与依赖就绪。
- 数据库连接失败时在模块导入阶段退出：会导致 liveness 不可访问，并产生导入副作用。
- Router 直接执行 SQL：违反项目分层原则。
