# ADR-0009 使用邮箱和密码认证

## 状态

已接受（2026-07-17）

## 背景

原 User Model 同时包含 `username`、`email` 和 `phone`，但 Sprint 1 没有用户名或手机号登录需求。要求用户同时维护 username 和 email，会增加注册输入、唯一性冲突和账号找回的复杂度。

项目未来需要通过邮箱接收验证码，但当前 Sprint 仍需学习密码哈希、密码验证和 JWT。使用邮箱验证码直接登录会引入验证码存储、TTL、限流和防刷等额外基础设施，改变 Sprint 1 的学习边界。

## 决策

- 使用 email 和密码完成注册、登录。
- `email` 是唯一登录标识，数据库中设置唯一约束。
- 不保留 `username` 和 `phone` 字段。
- `nickname` 是可选、非唯一的展示名称，不参与认证。
- 保留 `password_hash`，数据库永远不保存原始密码。
- 邮箱验证码在注册流程开始前单独设计，不属于 Story 1.1。
- `avatar_url` 允许为空；默认头像由客户端处理，用户上传头像未来存入对象存储。

## 原因

- 单一登录标识可以降低注册和账号冲突处理复杂度。
- 邮箱唯一约束同时保证业务唯一性和查询性能。
- 保留密码认证符合当前 Sprint 对 bcrypt 和密码安全的学习目标。
- 将验证码基础设施推迟到明确 Story，可以避免提前引入 Redis 或临时的不可靠存储。

## 影响

- Register 和 Login Schema 不包含 `username`。
- User Repository 只需要按 email 查询，不实现 `get_by_username()`。
- Service 在持久化前需要统一规范化 email。
- 后续加入邮箱验证时，可以扩展账号状态和验证流程，不需要恢复 username。

## 未采用方案

- username 和 email 同时作为身份字段：增加重复输入和唯一性冲突，没有当前业务价值。
- 邮箱验证码直接登录：需要提前引入验证码生命周期、限流和防刷设计，超出当前 Story 范围。
